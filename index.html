The Developer

Ever since we got our first computer from RadioShack way back in...probably, 1996, I've been interested in computers and how they work. I remember taking apart our early computers and installing new video cards so that they could run new games, like © Dark Age of Camelot, which was so much fun. Just the feeling of opening the CPU tower, installing that new graphics card, and getting it to work correctly was a great one; it felt like I was "hacking" my system!

I took my first coding class in ninth grade. We did some pretty basic stuff with languages like Pascal, C++, and Visual Basic (haha...see what I did there?!?). The feeling of getting those little programs to work correctly was ten fold the satisfaction as the graphics card installation. I wrote something and the computer behaved like it was supposed to; now that was really cool.

Regrettably, the main reason why I didn't go to college for a CS degree was...the Math. I was afraid. I looked at the prospectus for a Bachelor's in CS at LSU and thought "Wow, that's a lot of math" and made a 180° turn to, of all things, classical languages (mostly Latin).

It wasn't until many years later that I realized that the skills I gained in learning Latin, which is a complicated language in and of itself (it takes some pretty higher-order analytical skills to break down and make sense of a large chunk of literature from Cicero or Suetonius), could be translatable to coding. Breaking down larger sets of problems into smaller ones. Figuring out how the pieces of whatever entity fit together to produce the whole.

When I found out about the Coding Bootcamp at UT Austin, I thought "Well, I've learned Latin; I've learned Spanish; Javascript is just another language, with a bunch of syntax, and the analytical side of Latin will directly relate to learning Javascript...this sounds perfect!" So I decided to take the leap. As of writing this (7/5/2016), we are about halfway through and I've loved the experience so far. Sure, I definitely could have learned the curriculum on my own from the Internets, but I've really gotten a lot out of the guidance that my instructors, the TAs, and my classmates have offered and I wouldn't give that back for anything.

© Friend Finder

This was the first project I complete that was truly full-stack, using a back-end, routing, and a polished front-end. One of the trickiest things about this application was the algorithm for how to compare an array of answers from one user (the newest user) with the arrays of answers from each existing user, and then find the best match.

My approach to solving this was to take a step back, write out the process in pseudo-code (which really is how you should always start any project), and figure out how to achieve the deisred result along each step of the process. Once I had the concept in mind of what I was trying to achieve, it was just a matter of doing things in the correct order.

Overall I am very pleased with how this application turned out. I learned quite a lot throughout this process and came to understand the power behind Node and all the npm packages you can add on. Take a look at the final product!

  • Node
  • Express (with modularized endpoints)
  • Vex Modals
  • Form Validation (Javascript & jQuery)
  • HTML5
  • CSS3

PickUp

(Collaborative Project)

This was my first opportunity to work on a truly collaborative project and the learning curve was difficult, especially with regard to workflow and using Git. We as a team divided up the work between front-end (HTML and CSS) and behind-the-curtain (Javascript, APIs). I learned a lot of new stuff during the two weeks we spent working on this projects, including RegEx, form validation, making API calls utilizing AJAX and using the returned data appropriately, and persistent data storage using Firebase. My team and I worked very well together, never butting heads or having major disagreements about aspects of the project. I feel that we presented a great final product and that it could be easily scaled to include more locations, more activities, and members.

Technologies used for this project:

  • Javascript (form validation functions with RegEx)
  • jQuery (event listeners, hiding and showing DOM elements based on class)
  • AJAX (Google Maps Embed API, Weather Underground API)
  • Firebase (persistent data storage)
  • HTML5
  • CSS3

LIRI - Node App

This was my first venture into using Node.js and I enjoyed it much more than I thought I was going to. My expectations for my own satisfaction were low, but I ended up having a great time and was even able to work in an extra feature using the Inquirer npm. I was able to fetch and use quality data from the Spotify npm, the OMDB npm, and the Twitter npm. Digging through the vast JSON response from Spotify was a challenge, but it was a great moment when I finally figured it out, which was made much easier by using Postman.

Technologies used for this project:

  • Node.js
  • Inquirer npm (to control user input flow within the console)
  • Twitter npm
  • Spotify npm
  • OMDB npm
  • Node's built-in 'fs' package

Train Schedule

This was a fun app for many reasons. I hadn't ever really worked with append and prepend before, especially via Firebase, and dynamically populating a DOM element (a table, specifically) based on the data stored in the Firebase object. I also made clever use of Javascript's '%' modulo operator to calculate the time until the next train would hypothetically arrive. I also made use of Moment.js to help calculate train times in hours and minutes.

Technologies used for this project:

  • Javascript (form validation functions with RegEx)
  • jQuery (event listeners, event delegation [for dynamically added DOM elements], built-in methods)
  • Firebase (persistent data storage)
  • Moment.js (for calculating time)
